AI Agent Developer Brief: Fresh Platform
Platform Overview: Fresh is a multi-tenant SaaS operating system and consumer marketplace for salons. The stack uses the Next.js App Router (/app), centralizes business logic in /lib, and provides an internal API layer via /app/api.
1. Naming Conventions & Architecture
Database Tables & Keys: Tables use plural, snake_case naming (e.g., users, booking_services, platform_fees). Primary keys are consistently named id. Foreign keys follow the entity_id pattern (e.g., salon_id, client_id, staff_id).
Routing: Backend endpoints follow RESTful conventions within the Next.js App Router format, typically stored in route.js files (e.g., /api/admin/salons/route.js).
Components: Frontend React components use kebab-case naming conventions with a .jsx extension (e.g., booking-form.jsx, staff-creation-wizard.jsx) and are housed in the /components directory.
2. The 5 User Roles & Entity Mapping
Role enforcement utilizes stateless JWTs and symmetrical checks (e.g., requireRole(allowedRoles)). The 5 conceptual roles map to the database as follows:
Admin: Global platform administrators. They bypass standard ownership checks via a fast-return pattern: if (role === 'admin') return true;. This role is defined in the users table ENUM.
Owner: A salon's primary stakeholder. Their global users.id maps directly to the owner_id column in the salons table, granting cascading read/write access to all child business data.
Manager: A salon-specific proxy for the Owner. Crucial Distinction: "Manager" is not a global account role in the users table; it is an employment role defined within the staff table ENUM (role = 'manager').
Staff: Standard service providers. Mapped via users.id to a staff record, enabling them to view their specific schedule (bookings.staff_id) and manage their timesheets.
Client: Consumers booking services via the Widget or Marketplace. They are registered in the users table as role = 'client' and are atomically linked to businesses via the salon_clients relationship table during booking.
3. Strict 'Soft Delete' Rules
The platform strictly forbids permanent data destruction to maintain audit trails and financial integrity.
Standard Entities: Most core database entities (e.g., salons, bookings) rely on a deleted_at timestamp column to indicate soft deletion.
Relationship Tables: Mapping tables, specifically salon_clients, do not use timestamp columns; instead, they rely on an is_active boolean column (tinyint(1)) to handle soft deletions.
CRITICAL SCHEMA DISCREPANCY (Users): The core business logic specifies that user accounts should be soft-deleted by reassigning their role to 'deleted' (using role != 'deleted' in SQL queries). Do not execute this logic. The actual database schema defines users.role strictly as ENUM('client','owner','staff','admin') and lacks a deleted_at column. Attempting to set a role to 'deleted' will cause a strict SQL mode error. You must either create a migration to add a proper deleted_at column to the users table, or add 'deleted' to the ENUM before implementing user soft deletes.
4. Core Operation Rules
Booking Safety: All bookings must route through createSafeBooking() in /lib/booking.js. This enforces strict database transaction controls using SELECT ... FOR UPDATE to lock rows and prevent concurrent double-booking of staff.
Checkout & Financials: Never trust frontend math. Checkout totals must be calculated purely from backend database aggregates: SUM(services) + SUM(products) - SUM(discounts) - SUM(gift_cards).
Client Deduplication: Client creation must use findOrCreateClient(), which relies on SELECT ... FOR UPDATE for phone-first resolution and explicitly catches ER_DUP_ENTRY errors generated by the uq_users_email constraint to prevent phantom duplicate accounts
